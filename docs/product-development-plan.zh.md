# iCloudFriend 产品开发指导

更新日期：2026-06-15

本文基于《iCloudFriend 商业计划书》（`docs/iCloudFriend商业计划书.pptx`）与当前仓库实现整理，用来指导后续产品和工程开发。核心原则是：先把“可靠完成本地原片备份”做成用户可独立成功的闭环，再逐步扩展付费能力、智能管理和增长体系。

## 1. 产品定位

iCloudFriend 是面向 iPhone 用户的本地优先照片备份工具。它把 iPhone / iCloud Photos 中的原始照片、视频、Live Photo 配对资源和元数据，通过局域网备份到用户自己的 Windows 电脑或 NAS 目录。

北极星目标：

让普通用户在 15 分钟内完成首次配置，并能确认自己的照片已经以原始质量、安全、可恢复的方式备份到本地。

当前最应该坚持的三条价值主张：

- 便捷：不依赖数据线和复杂专业工具，Windows 端负责准备共享目录，iPhone 端负责选择目标文件夹并启动备份。
- 经济：核心备份能力免费，后续用高级管理能力支撑一次性买断。
- 隐私安全：照片不进入 iCloudFriend 自有服务器，备份目标由用户自己控制。

## 2. 商业计划书提炼

计划书提出的关键承诺可以拆成四类：

| 类别 | 计划书承诺 | 对开发的含义 |
| --- | --- | --- |
| 核心备份 | 高速 Wi-Fi、本地传输、原图无损、断点续传、智能增量 | 先保证备份可靠性、完整性和可验证性，再优化速度 |
| 格式完整性 | HEIC、RAW、Live Photos、EXIF/GPS 元数据完整保留 | 需要覆盖文件资源、sidecar metadata、恢复/导出验证 |
| 高级管理 | AI 分类、人物/地点/场景标签、重复照片清理、HEIC 转换 | 应作为 Pro 版第二阶段能力，不要阻塞 MVP |
| 商业化 | 基础版免费、专业版 99 元买断、基础版每月 500 张限制 | 需要授权、额度、支付、权益恢复、离线校验等系统支持 |

需要校准的表述：

- 当前项目实现路径是 iOS Files Provider + SMB/NAS 目录，不是自研无线协议。外部表达建议使用“局域网备份到 Windows/SMB 共享”，避免承诺“自动直连”。
- 计划书中同时出现“高速 Wi-Fi”和“USB 3.0 极速传输”。当前代码没有 USB 传输能力，短期应统一为 Wi-Fi / SMB。
- 如果照片原片不在本机，iOS 会从 iCloud 下载原始资源后再写入 SMB。营销上应说明“照片传输到电脑的过程不上云，但下载 iCloud 原片可能需要联网”。
- 当前没有应用层端到端加密。可以说“数据不经过我们的服务器”，暂时不要承诺“点对点加密”或“银行级加密”，除非后续实现并验证。
- 市场规模、速度 XXMB/s、备份 100GB 耗时、58% 隐私担忧等数据需要补来源或通过测试替换。

## 3. 当前实现对照

已经具备的能力：

- iOS 端全量/增量备份流程：`ios/iCloudFriend/iCloudFriend/Services/BackupManager.swift`
- Photos 原始资源枚举与导出：`AssetMetadataBuilder.swift`、`AssetResourceExporter.swift`
- `.partial` 文件续传与最终文件原子替换
- SHA-256 校验、metadata sidecar、events.ndjson
- Windows 端创建/修复 SMB share、展示连接信息、监控备份索引
- Windows smoke test 与基本构建说明
- 备份格式文档：`docs/backup-format.md`

尚未覆盖或只覆盖一部分的能力：

- 首次使用向导、错误诊断、权限/网络/SMB 失败自助修复
- 自动发现 Windows 电脑或自动配对
- 备份完成后的完整性验证报告
- 从备份恢复到电脑目录或导回 Photos 的流程
- AI 分类、去重、HEIC 转换、付费授权、额度限制
- 可发布版本的隐私政策、用户协议、崩溃/日志策略
- iOS 端自动化测试、真实设备回归矩阵、性能 benchmark

## 4. 阶段路线图

### P0：可靠备份闭环

目标：让技术早期用户在 README 指引下完成一次真实照片库备份。

必须完成：

- Windows 端创建 `iCloudFriend/Backup` 共享目录，并稳定展示 `smb://...` 地址。
- iPhone 端选择 `Backup` 文件夹后，可以完成 Full 和 Incremental 两种模式。
- 支持 JPEG、HEIC、MOV、Live Photos、RAW/DNG、隐藏/收藏、有地理位置的素材。
- 中断后再次运行不会重复写已完成文件，`.partial` 能继续或安全重置。
- Windows 端能显示 asset 数、resource 数、总大小、最近备份和 metadata 错误。

验收口径：

- 用至少 1,000 个混合素材完成首次备份。
- 人为中断网络或退出 app 后重试，最终 metadata 与资源文件可被 Windows 端索引。
- 抽样 20 个资源，metadata 中 SHA-256 与磁盘文件一致。
- 对“iCloud 原片需下载”的场景有清晰 UI 提示。

### P1：公开 Beta 可用性

目标：让非技术用户也能独立完成首次成功备份。

优先开发：

- Windows 首屏引导：选择备份盘、创建共享、复制地址、显示当前 Windows 用户名。
- iOS 首次向导：先在 Files 连接 SMB，再选择 `Backup` 文件夹，再开始备份。
- 失败诊断：Photos 权限、文件夹权限失效、SMB 不可写、磁盘空间不足、网络中断。
- 备份完成报告：本次新增资产、失败资产、耗时、平均速度、总大小、可验证校验结果。
- 日志导出包：用户遇到问题时可发送给开发者，日志不包含照片内容和敏感路径。

### P2：Pro 版基础

目标：让 99 元买断有明确价值，但不破坏免费用户的核心体验。

建议 Pro 功能：

- 多设备管理：按设备/用户区分备份来源，避免多台 iPhone 的 asset 命名和状态混乱。
- 高级校验：全库重扫、缺失文件修复建议、重复备份检测。
- Windows 照片浏览器：按日期、设备、地点、格式筛选本地备份。
- 恢复助手：把选中的备份资源导出到普通文件夹，或生成可人工导入 Photos 的恢复包。
- 自动计划提醒：提示用户定期备份，不做后台静默承诺。

暂缓放入 Pro 首版：

- AI 人物识别与场景识别。它价值高，但会引入模型、隐私、性能、跨平台和误判成本。
- 每月 500 张免费额度限制。它需要授权和计量体系，过早加入会增加摩擦。

### P3：智能管理与增长

目标：从“备份工具”升级为“本地照片资产管理工具”。

可探索方向：

- 本地 AI 标签：时间、地点、格式、事件、截图、文档、相似照片等低风险分类先行。
- 重复/相似照片清理：先提供建议与预览，不自动删除。
- HEIC 转 JPEG：作为导出功能，而不是破坏原始备份。
- NAS 兼容性优化：Synology、QNAP、Windows 共享、外置硬盘路径分别测试。
- 推荐激励与口碑传播：等产品留存和成功率稳定后再做。

## 5. 工程任务优先级

| 优先级 | 任务 | 为什么重要 | 参考位置 |
| --- | --- | --- | --- |
| P0 | 备份完成校验报告 | 把“已备份”从感觉变成可证明结果 | iOS `BackupManager`、Windows `backupIndex.js` |
| P0 | 失败原因结构化 | 当前日志偏技术，用户需要知道怎么修 | iOS `BackupError`、Windows share status |
| P0 | 真实文件类型测试集 | 商业计划强调无损格式，需要证据 | `docs/backup-format.md` |
| P1 | 首次向导 | 降低 SMB/Files Provider 的理解成本 | iOS `ContentView.swift`、Windows `renderer.js` |
| P1 | 性能 benchmark | 替换计划书中的 XXMB/s 和 XX分钟 | 新增 `docs/benchmarks.md` |
| P1 | 隐私与网络说明 | 避免“完全无需联网”被误解 | README、应用内文案 |
| P2 | 多设备状态模型 | 支撑 Pro 版和家庭场景 | metadata schema、Windows index |
| P2 | 恢复/导出助手 | 备份价值最终要靠恢复闭环证明 | Windows app |
| P3 | 本地智能分类 | 对应商业计划 AI 价值，但应后置 | Windows app / optional local model |

## 6. 产品验收指标

开发阶段不建议先追求用户量，先追求备份成功率和可解释性：

- 首次配置成功率：用户从打开 Windows app 到 iPhone 成功写入第一张照片的比例。
- 首次完整备份成功率：首次 Full sync 无人工干预完成的比例。
- 失败可恢复率：中断/权限/磁盘空间问题后，用户按提示恢复成功的比例。
- 备份完整性：metadata 数量、resource 数量、hash 校验通过率。
- 平均吞吐：按素材类型和网络环境记录 MB/s，建立真实速度口径。
- 支持工单成本：每 100 个用户产生的问题数，以及是否能通过日志定位。

## 7. 下一轮两周开发建议

建议先做一个“Beta 可信度冲刺”：

- 新增备份完成摘要：耗时、资源数、总大小、失败列表、平均速度。
- 新增 hash 抽样/全量验证入口，Windows 端展示校验状态。
- 梳理 iOS 错误枚举，把常见失败映射为用户可执行建议。
- 更新 README，明确当前是 SMB/Files Provider 模式，并解释 iCloud 原片下载。
- 建立真实设备测试表：iPhone 型号、iOS 版本、Windows 版本、网络、素材类型、结果。
- 跑通 Windows `npm run check` 和 `npm run smoke`，并记录发布前必跑命令。

## 8. 决策清单

这些问题会直接影响开发顺序，建议尽快定稿：

- 免费版是否真的限制每月 500 张，还是改为“核心备份免费，高级管理付费”。
- 专业版 99 元买断是否包含终身大版本升级；若包含，长期收入模型需要其他补充。
- 是否承诺应用层加密。如果承诺，需要设计密钥、恢复、跨设备和性能方案。
- 是否支持自动发现 Windows 端。如果支持，需要设计局域网发现协议和防火墙引导。
- AI 能力是否全部本地化。如果需要本地化，应优先选择低成本的规则/元数据分类。
- 是否把 NAS 作为一等公民。如果是，需要在文案、测试和 UI 中独立呈现 NAS 路径。

## 9. 推荐开发原则

- 不要为了增长提前牺牲备份可信度。照片备份工具的第一信任来自“不会丢、能恢复、能证明”。
- 保留原始资源永远优先于格式转换。转换只能是导出功能，不能替代原始备份。
- 所有商业计划中的数字先进入待验证状态，只有 benchmark 或可信来源支撑后再写入对外材料。
- 先把用户操作路径做短，再做 AI。对这个产品来说，最强的智能可能是让用户不必理解 SMB。
- 每一个付费功能都应该降低真实管理成本，而不是只增加功能列表长度。
